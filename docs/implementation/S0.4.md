# S0.4 – Documentation Templates (Implementation Plan)

*This is a living implementation document; future updates will be appended.*

## Objective
Design the official documentation template system for FastMin. The templates will serve as the standard starting point for all future milestones and architectural decisions, ensuring consistency, minimal duplication, and adherence to the architecture‑first, living‑documentation philosophy.

## Purpose of the Template System
- Provide a **single source of truth** for the structure of all milestone‑related documents.
- Enforce the **architecture‑first** approach by requiring an `architecture.md` before any implementation work.
- Keep documentation **consistent** across milestones while allowing each team to append specifics.
- Reduce repetitive effort when starting a new milestone or architectural decision record.

## Directory Structure
```
/docs/
├─ templates/                # <-- future location of all markdown templates
│   ├─ architecture.md       # High‑level design, constraints, and diagrams
│   ├─ implementation_plan.md# Detailed implementation plan for a milestone
│   ├─ implementation.md     # Actual implementation notes, code references
│   ├─ validation.md        # Test strategy, validation results, metrics
│   ├─ retrospective.md      # Post‑mortem notes, lessons learned
│   └─ adr.md                # Architecture Decision Records template
└─ implementation/
    └─ S0.4.md               # This implementation plan (living doc)
```
The `templates/` directory will be version‑controlled alongside the codebase so that changes to templates are tracked.

## Relationship with Key Documents
- **`CONSTITUTION.md`** – The template system operationalises the Constitution’s principles (Open Collaboration, Living Documentation, Architecture‑first). Every template includes a reference to the Constitution to remind authors of the guiding philosophy.
- **`WORKFLOW.md`** – The workflow’s gates (Architecture Review → Implementation Plan Review → Validation) map directly to the corresponding templates (`architecture.md`, `implementation_plan.md`, `validation.md`). The templates make it easy to produce the artefacts required at each gate.
- **Implementation documents** – Each milestone’s implementation document (e.g., `S0.4.md`) will *append* a progress log for the template system itself, mirroring the approach used for previous milestones.

## Future Extensibility
- New templates can be added under `templates/` without affecting existing ones; the directory layout is deliberately flat to keep discovery simple.
- Template files can evolve (e.g., adding sections) by **appending** a version‑specific changelog at the bottom, preserving historic content.
- The system can be reused for non‑milestone artefacts (e.g., feature tickets) by copying the appropriate template.

## Risks & Mitigations
| Risk | Impact | Mitigation |
|------|--------|------------|
| Over‑complex templates discourage use | Low adoption, fragmented docs | Keep templates minimal; enforce a lightweight “append‑only” discipline |
| Template drift (templates diverge from workflow) | Inconsistent documentation | Periodic review during Architecture Review meetings; version the templates |
| Missing templates for new processes (e.g., security review) | Gaps in documentation | Allow easy addition of new templates; schedule a template‑review sprint |

## Template Descriptions
### 1. `architecture.md`
- **Purpose:** Capture the high‑level design, constraints, diagrams, and key decisions *before* any code is written.
- **When used:** At the start of every milestone after the idea phase.
- **Who uses it:** Architects, Technical Lead, and Product Owner.
- **Typical sections:** Goal, Context Diagram, Component Overview, Interfaces, Non‑functional Requirements, Risks, Open Questions.
- **Relationships:** Feeds directly into `implementation_plan.md` and is required for the Architecture Review gate.

### 2. `implementation_plan.md`
- **Purpose:** Detail the concrete steps to turn the architecture into code, including tasks, dependencies, and verification plan.
- **When used:** After architecture approval, before any implementation begins.
- **Who uses it:** Contributors and Technical Lead.
- **Typical sections:** Overview, Task Breakdown, Dependencies, Verification Plan, Verification Steps, Success Criteria.
- **Relationships:** Links to `architecture.md` (reference) and to `implementation.md` (later detailed notes).

### 3. `implementation.md`
- **Purpose:** Record the actual implementation work, code locations, design rationales, and any deviations from the plan.
- **When used:** During the Implementation phase.
- **Who uses it:** Contributors.
- **Typical sections:** Files Modified, Code Highlights, Rationale, Issues Encountered, Temporary Work‑arounds.
- **Relationships:** Mirrors the tasks in `implementation_plan.md` and serves as input for `validation.md`.

### 4. `validation.md`
- **Purpose:** Document testing strategy, results, performance metrics, and validation against requirements.
- **When used:** After implementation, before the Validation Review gate.
- **Who uses it:** Test engineers, Contributors.
- **Typical sections:** Test Plan, Test Cases, Results, Metrics (RAM, Flash, Power), Issues, Pass/Fail Summary.
- **Relationships:** References `implementation.md` for traceability and feeds into `retrospective.md`.

### 5. `retrospective.md`
- **Purpose:** Capture lessons learned, what worked well, what can be improved, and new backlog items.
- **When used:** After a milestone is closed (post‑release or post‑cleanup).
- **Who uses it:** Whole team during the Retrospective meeting.
- **Typical sections:** Successes, Challenges, Action Items, New Backlog Items.
- **Relationships:** Closes the loop for the milestone and informs future `architecture.md` decisions.

### 6. `adr.md` (Architecture Decision Record)
- **Purpose:** Provide a lightweight, versioned record of significant architectural decisions.
- **When used:** Whenever a major design choice is made that impacts the system.
- **Who uses it:** Architects and Technical Lead.
- **Typical sections:** Title, Status, Context, Decision, Consequences, Related ADRs.
- **Relationships:** Often referenced from `architecture.md` and from later `implementation_plan.md` when decisions affect implementation.
