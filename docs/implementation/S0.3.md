# S0.3 – Contributing Guide (Implementation Plan)

*This is a living implementation document; future updates will be appended.*

## Objective
Define the implementation plan for the project’s **Contributing Guide** (`docs/CONTRIBUTING.md`).
The guide will help new and existing contributors understand how to prepare changes, follow coding and documentation standards, and navigate the milestone workflow.

## Sections to be covered in `docs/CONTRIBUTING.md`

1. **Purpose**
   - Explain why a CONTRIBUTING guide exists: to lower the entry barrier, ensure consistent quality, and streamline collaboration.

2. **Target Audience**
   - New contributors (first‑time developers, students, hobbyists)
   - Experienced contributors (maintainers, reviewers)
   - External partners who may submit patches or issues.

3. **Relationship with `CONSTITUTION.md`**
   - The guide reinforces the engineering principles defined in the Constitution (Open Collaboration, Living Documentation, Semantic Versioning, etc.).
   - Any deviation from those principles must be justified and documented.

4. **Relationship with `WORKFLOW.md`**
   - Mirrors the phases of the official workflow (Architecture → Release).
   - Provides concrete actions contributors must take at each gate (e.g., submit a design doc before Implementation Plan, obtain review approval before merging).

5. **Repository Expectations**
   - Branch naming conventions (`feature/<name>`, `bugfix/<name>`, `doc/<name>`).
   - Keep the `main` branch deploy‑ready; use feature branches for work.

6. **Coding Standards**
   - Formatting (clang‑format/`arduino-lint` settings).
   - Naming conventions (snake_case for files/variables, PascalCase for types).
   - Use of `constexpr`, `static`, and value semantics where appropriate.
   - No dynamic allocation unless absolutely required.

7. **Documentation Expectations**
   - All code changes must be accompanied by updates to inline comments and relevant markdown docs.
   - New public APIs require a README entry or section in the appropriate module doc.
   - Follow the same “open → append → save” policy for all docs.

8. **Pull Request Expectations**
   - PR title: concise, prefixed with the appropriate tag (`feat:`, `fix:`, `docs:`).
   - Description must include: purpose, related issue, testing performed, and any required documentation updates.
   - Must pass CI checks (build, lint, unit tests).

9. **Review Process**
   - At least one **Reviewer** must approve each gate (Architecture Review, Implementation Plan Review, Validation).
   - Reviewers check compliance with the Constitution, Workflow, and Coding Standards.

10. **Commit Message Conventions**
    - Follow Conventional Commits (`type(scope): description`).
    - Types: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`.
    - Include a brief footer with “Closes #<issue>” when applicable.

11. **Future Extensibility**
    - The guide will evolve as new phases (e.g., Security Review) are added to the workflow.
    - Section placeholders will be added for future topics (e.g., “Testing Strategy”, “Performance Guidelines”).

12. **Success Criteria**
    - New contributors can open their first PR without needing direct mentor assistance.
    - All merged PRs comply with the coding standards and pass CI.
    - Documentation updates are present for every code change.
    - Review turn‑around time meets the project’s SLA (e.g., 48 h for standard PRs).
